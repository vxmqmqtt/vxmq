# 协作规范

本文档定义项目参与者之间的默认协作规则。若后续有补充，以最新版本为准。

## 基本规则

1. 后续交互输出结果统一使用中文。
2. 任何实现前，先明确目标、边界、输入输出和验收标准；目标不清时先澄清，不直接编码。
3. 对较大的改动，先给方案再实施；对小改动，可直接实施并说明原因与影响。
4. 所有重要信息默认沉淀到 `docs/`，避免只停留在对话中。
5. 涉及 MQTT 协议语义时，以协议正确性优先，不以实现便利替代规范要求。
6. 对不确定信息必须显式标注为“已确认”“待确认”或“假设”。
7. 每项重要功能都应同步考虑测试策略与验收方式，避免先写实现、后补验证。
8. 若发现当前方案会造成明显返工，应直接指出并给出替代方案。
9. 重要架构或范围决策应形成书面记录，必要时进入 `07-project/decisions/`。
10. 每次阶段性工作完成后，应说明结果、影响范围和建议的下一步。
11. 项目目标是同时支持 MQTT 3.1.1 与 MQTT 5；新增设计或实现不得默认只面向单一版本，除非文档明确说明阶段性限制。
12. 对协议未规定、允许多种实现或存在 `MAY` 行为的地方，实现前应先核对 OASIS 规范，再参考 HiveMQ、EMQX、Mosquitto 等同类产品的官方资料或实测行为，并将结论记录到 ADR 或相关设计文档。
13. `M1` 主链路默认按响应式、非阻塞、event-loop 模型实现；若需要引入阻塞操作，必须先显式设计边界并记录原因。
14. 在 Quarkus 中直接接入 Vert.x 扩展时，应先检查是否存在 Mutiny 变体；若存在，默认优先使用 Mutiny 变体，只有在没有包装层或包装层能力不足时才使用原生 Vert.x 类型。

## 文档规则

- 新增文档前先判断是否已有合适位置，避免重复。
- 文档标题应直接表达主题，减少“设计说明一”“方案草案二”这类命名。
- 同一主题若既有需求又有设计，应拆分到对应目录并互相链接。
- 若文档内容依赖假设条件，需在文首或章节内明确说明。

## 开发规则

- 优先构建最小闭环，再扩展高级能力。
- 优先可验证实现，再做性能优化。
- 对关键行为应建立自动化测试，而不是依赖手工验证。
- 变更应尽量局部化，避免在需求尚不明确时进行大范围重构。
- 所有新增或修改的生产代码、测试代码都必须补充注释，不允许提交“无注释代码”。
- 注释遵循 Java 一般最佳实践：公共类型、接口和关键方法使用简洁 Javadoc 说明职责、核心输入输出或协议语义；复杂状态转换、协议分支、并发处理和回滚逻辑使用块注释解释“为什么这样做”。
- 注释必须与代码保持同步，禁止写与代码字面重复的废话式注释，例如“给变量赋值”“调用方法”这类低信息量描述。

## 沟通规则

- 讨论问题时优先给出结论、依据和风险，而不是只给观点。
- 如果存在多个可行方案，应说明取舍，不默认把复杂方案当成更优方案。
- 若需要用户确认，应尽量把问题收敛成少量关键决策点。
