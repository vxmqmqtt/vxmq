# ADR-0003 M1 阶段的重叠订阅投递策略

- 状态：已接受
- 日期：2026-03-14

## 背景

当同一客户端存在多个非共享订阅，且这些 Topic Filter 同时命中同一条消息时，MQTT 规范没有强制要求必须投递多份消息。

OASIS MQTT 3.1.1 明确指出：

- 服务端必须至少向客户端投递一份消息，并遵守所有命中订阅中的最大 QoS。
- 服务端还可以为每个额外命中的订阅继续投递额外副本。

这意味着“单次投递”与“多次投递”都在规范允许范围内，属于实现选择。

在同类产品官方资料中，HiveMQ 对 Subscription Identifier 的说明显示：当一个出站 PUBLISH 同时命中同一客户端的多个订阅组合时，会在同一个出站 PUBLISH 中附加多个 Subscription Identifier。这至少表明“单个出站 PUBLISH 承载多个匹配结果”是现实存在的实现路径。

## 决策

`M1` 对非共享重叠订阅采用“同一客户端单次投递”策略：

1. 同一客户端的多个 Topic Filter 命中同一 Topic Name 时，只向该客户端发送一份消息。
2. 该份消息的投递 QoS 取所有命中订阅允许的最大值与原始发布 QoS 的较小者。
3. `M1` 当前只支持 QoS 0，因此实现阶段可先简化为“去重后单次 QoS 0 投递”。

## 原因

- 该策略完全落在规范允许范围内。
- 可以显著简化 `routing` 与 `publish` 模块的实现。
- 便于后续引入 Subscription Identifier、共享订阅和 QoS 1 / QoS 2 时逐步扩展。

## 结果

- `topic-routing.md` 中的查找结果需要按客户端去重。
- `publish-flow.md` 中的投递规则以“每客户端一份消息”为准。
- 后续如需切换为“多副本投递”，必须通过新的 ADR 明确变更。

## 局限

- 当前未找到 HiveMQ / EMQX / Mosquitto 官方文档对“非共享重叠订阅”是否默认多副本投递的直接统一说明。
- 后续互操作性测试阶段应继续补充对同类产品行为的实测确认。
