# 协议兼容性要求

本文档定义 `vxmq` 的协议兼容性目标，以及协议决策的优先级。

## 目标

`vxmq` 以严格遵循 MQTT 协议为前提，同时支持以下版本：

- MQTT 3.1.1
- MQTT 5

说明：

- 当前项目规划仍以 MQTT 5 特性矩阵驱动功能展开，但这不意味着 MQTT 3.1.1 可以被忽略。
- 任何核心链路设计都应检查其在 MQTT 3.1.1 和 MQTT 5 下的差异。

## 兼容性原则

1. 优先遵循 OASIS MQTT 规范，而不是模仿单个产品的历史行为。
2. 对 MQTT 3.1.1 与 MQTT 5 的共同语义，应尽量复用统一实现。
3. 对版本差异，应显式建模，不用“隐式 if 分支”散落到各处。
4. 对协议未规定、实现自由度较高或存在 `MAY` 的行为，在决策前先核对同类产品的官方资料或实测行为。
5. 若同类产品行为不一致，最终选择必须通过 ADR 固化。

## 决策优先级

遇到协议行为决策时，按以下顺序处理：

1. OASIS MQTT 规范
2. 已接受的项目 ADR
3. 同类产品官方资料或实测结果
4. 项目内部新决策与补充 ADR

## 同类产品参考范围

优先参考以下 Broker：

- HiveMQ
- EMQX
- Eclipse Mosquitto

约束：

- 优先使用官方文档、官方配置说明、官方行为说明。
- 若官方文档不足，可在后续互操作性测试阶段补充实测结论。
- 引入同类产品行为时，要区分“协议要求”与“实现选择”。

## 当前已确定的兼容性决策

- 支持 MQTT 3.1.1 与 MQTT 5。
- 连接阶段的空 `clientId` 处理遵循版本差异化策略，见 `07-project/decisions/0002-m1-client-id-policy.md`。
- 非共享重叠订阅命中同一客户端时，`M1` 先采用单次投递策略，见 `07-project/decisions/0003-m1-overlapping-subscription-delivery.md`。

## 后续工作要求

- 新增协议流程文档时，必须标注是否存在 MQTT 3.1.1 / MQTT 5 差异。
- 引入新的协议自由度决策时，必须补 ADR。
- 互操作性测试阶段应至少覆盖 HiveMQ、EMQX、Mosquitto 中的常见客户端或 Broker 行为比对。
