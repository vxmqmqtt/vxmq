# MQTT 5 特性矩阵

本文档用于回答三个问题：

1. 要做哪些 MQTT 5 能力。
2. 每项能力计划在哪个阶段落地。
3. 何时可以认为该能力“真正完成”。

## 状态定义

- `未开始`: 尚未进入设计或实现。
- `设计中`: 已有明确设计方向，但未完成实现。
- `已实现`: 功能已编码完成，但验证不充分。
- `已验证`: 功能已通过对应测试与验收。

## 阶段定义

- `M1`: 最小可连接、可订阅、可发布的单机 Broker 闭环。
- `M2`: 补齐会话与消息可靠性语义，形成可用核心。
- `M3`: 补齐 MQTT 5 关键属性与基础安全、观测能力。
- `M4`: 强化持久化、恢复能力、互操作性和性能基线。
- `M5`: 补齐高级特性与生产化增强项。

## 验收等级

- `A`: 单元测试通过，核心分支有覆盖。
- `B`: 集成测试通过，包含端到端报文交互。
- `C`: 互操作性验证通过，至少与一个外部 MQTT 客户端稳定互通。
- `D`: 异常路径或边界条件得到验证。

默认要求：

- 核心协议特性至少达到 `A + B + D`。
- 对外可见行为或兼容性敏感特性，要求达到 `A + B + C + D`。

## 里程碑出口标准

### M1 出口

- 客户端可稳定完成连接、订阅、发布、取消订阅、断连。
- QoS 0 主链路可用。
- Topic Filter 与通配符语义正确。
- 关键交互具备自动化集成测试。

### M2 出口

- 会话恢复、Session Expiry、Retained Message、Will Message 可用。
- QoS 1 / QoS 2 状态机语义正确。
- Broker 在异常断连与重连场景下行为可预测。

### M3 出口

- MQTT 5 关键属性完成首轮支持。
- 基础鉴权、健康检查、指标、日志具备可用实现。
- 协议问题定位与运行状态观察具备基础手段。

### M4 出口

- 明确持久化策略并完成首轮恢复能力。
- 互操作性测试与性能基线测试形成固定流程。
- 至少形成“可以验证稳定性”的工程闭环。

### M5 出口

- 高级特性完成补齐或明确延后策略。
- 针对生产化能力的主要缺口有明确结论。

## 使用规则

- 每次状态变更时，应同步补充关联测试、缺口或阻塞项。
- 若某项特性被拆分，应在备注中写清边界，避免“部分实现”被误判为完成。
- 若阶段计划调整，应同步更新 `07-project/milestones.md`。
- 若某项特性暂时降级实现，应显式记录降级范围和影响。
